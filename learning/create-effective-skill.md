# 创建 Skill 的核心思路

来源：Anthropic "Building Effective Agents" + Block Engineering 设计原则 + Anthropic Skill 最佳实践

## 核心思想：用「宪法」替代「SOP」

不要告诉 Agent「怎么做」（步骤），而是告诉 Agent「什么不能做」和「什么必须做」（约束）。

固定 SOP 会让 Agent 在判断错误时"一条道走到黑"。约束式设计划定边界的同时，保留 Agent 根据上下文灵活决策的空间。

---

## 五个设计原则

### 1. 二区架构：区分「一致性」和「灵活性」

```
一致性区域 → 用脚本/硬规则（相同输入，相同输出）
灵活性区域 → 交给 Agent（每个 case 都不同）
```

- 需要一致性的：评分逻辑、验证规则、必须遵守的约束
- 需要灵活性的：执行顺序、问题诊断路径、内容生成

### 2. 约束 > 步骤

给 Agent 思考的维度，而非固定的流程。

```
❌ "先检查 A，再检查 B，最后修复"
✅ "排查维度：配置、代码、知识、验证（按需选择）"
```

### 3. 写宪法，不写建议

- 越具体越好，明确约束边界
- 灰色地带用 Agent 自己判断，但红线不可逾越

### 4. 内置自我纠错机制

在 Skill 中明确写出「预警信号」：

- 修复方案没有生效
- 知识库结果与假设矛盾
- 用户反馈「还是不行」
- 超过 N 次尝试仍未解决

触发预警时，Agent 应重新评估判断，而非重复无效操作。

### 5. 简约至上 + 渐进式披露

- SKILL.md 只放入口点和核心约束（~100 行以内）
- 详细内容通过子文件按需加载
- 只添加 Agent 真正需要的上下文

---

## Anthropic 三条核心原则

1. **保持简洁** — 最简单的能工作的方案就是最好的方案
2. **保持透明** — 让 Agent 展示推理过程
3. **精心设计接口** — Agent 与工具/系统的交互方式值得投入和 prompt 工程一样多的精力

---

## 一句话总结

好的 Skill = 清晰的约束（宪法）+ 灵活的空间（Agent 自主决策）+ 自我纠错机制（预警信号）

# Harness Engineering: 利用 Codex 构建 Agent-First 世界

> 来源: https://openai.com/index/harness-engineering/
> 阅读日期: 2026-03-20
> 作者: OpenAI 团队

## 一、实验背景

### 1.1 实验目标

OpenAI 团队在过去 5 个月做了一个大胆的实验：**从零行手写代码开始，完全由 AI Agent（Codex）构建一个软件产品**。

这个产品：
- 有内部日常用户
- 有外部 alpha 测试者
- 会正常部署、崩溃、然后被修复

### 1.2 核心约束

团队故意选择了这个约束：**0 行手写代码**

目的：构建能够将工程效率提升数量级所需的一切。

时间紧迫：几周内交付最终 100 万行代码的产品。

### 1.3 关键数据

| 指标 | 数值 |
|------|------|
| 初始提交 | 2025年8月底 |
| 最终代码规模 | 约 **100 万行** |
| PR 数量 | 约 **1500 个** |
| 初始团队规模 | 3 名工程师 |
| 最终团队规模 | 7 名工程师 |
| 吞吐量 | **3.5 PRs/工程师/天** |
| 开发时间节省 | 约 **90%** |

**核心理念：Human steer, Agents execute. （人类掌舵，Agent 执行）**

## 二、工程师的新角色

### 2.1 角色转变

当 Agent 承担主要编码工作时，工程师的职责发生了根本变化：

**不再是写代码**，而是：
- **设计环境** (design environments)
- **表达意图** (specify intent)
- **构建反馈循环** (build feedback loops)

### 2.2 "不写代码"的工程工作

由于缺乏亲身编码，团队引入了不同类型的工程工作：
- 系统设计
- 脚手架搭建
- 杠杆效应（leverage）构建

## 三、经验教训

### 3.1 教训一：环境需要深度工作

**问题**：早期进展比预期慢，不是因为 Codex 能力不足，而是**环境描述不足**。

Agent 缺乏：
- 必要的工具
- 抽象层级
- 内部结构

这些是实现高级目标所必需的。

**解决方式**：深度优先工作法
1. 把大目标分解成小构建块（设计、代码、审查、测试等）
2. 引导 Agent 构建这些构建块
3. 用它们解锁更复杂的任务

**关键洞察**：当失败时，修复方案**从来不是"更努力"**。因为取得进展的唯一方式是让 Codex 工作，人类工程师总是介入并问：
> "**缺少什么能力？我们如何让它对 Agent 来说既清晰又可执行？**"

### 3.2 教训二：人类通过 Prompt 交互

**工作流程**：
1. 工程师描述任务
2. 运行 Agent
3. Agent 打开 PR

**PR 完成流程**：
- 让 Codex 本地审查自己的更改
- 请求额外的特定 Agent 审查（本地和云端）
- 响应任何人类或 Agent 反馈
- 循环迭代直到所有 Agent 审查者满意

> 这被戏称为 ["Ralph Wiggum Loop"](https://ghuntley.com/loop/)

Codex 直接使用标准开发工具：
- gh CLI
- 本地脚本
- 仓库嵌入的 skills

**无需人类复制粘贴到 CLI。**

### 3.3 教训三：PR Review 可以完全由 Agent 完成

- 人类可以审查 PR，但不是必须的
- 逐渐将几乎所有审查工作推给 Agent-to-Agent

### 3.4 教训四：让应用对 Agent 可读

随着代码吞吐量增加，瓶颈变成了**人类 QA 容量**。

解决方式：让应用本身对 Codex 可读：
- **UI 可读**：通过 Chrome DevTools Protocol
- **日志可读**：Wire 日志到 Agent 运行时
- **指标可读**：应用指标直接暴露给 Agent

**具体实践**：
- 应用可以通过 git worktree 启动，让 Codex 可以为每个更改启动一个实例
- 接入 Chrome DevTools Protocol，创建处理 DOM 快照、截图、导航的 skills
- 这使 Codex 能够：重现 bug、验证修复、直接推理 UI 行为

### 3.5 教训五：可观测性工具对 Agent 开放

**本地可观测性栈**：
- 日志、指标、追踪通过本地可观测性栈暴露给 Codex
- 对任何给定 worktree 是临时的
- Codex 在完全隔离的版本上工作（包括日志和指标）
- 任务完成后日志和指标被销毁

**Agent 可以**：
- 用 LogQL 查询日志
- 用 PromQL 查询指标

**示例 Prompt**：
- "确保服务启动在 800ms 内完成"
- "确保这四个关键用户旅程中没有超过 2 秒的 span"

### 3.6 长时间运行

**观察到**：单个 Codex 运行可以在单个任务上工作 **6 小时以上**（通常在人类睡觉时）

## 四、上下文管理

### 4.1 最大挑战

上下文管理是让 Agent 有效处理大型复杂任务的**最大挑战之一**。

### 4.2 教训：给地图，不要给 1000 页手册

团队尝试了"**一个大 AGENTS.md**"方法，结果**可预测地失败了**：

| 问题 | 说明 |
|------|------|
| 上下文是稀缺资源 | 巨大的指令文件挤占了任务、代码和相关文档 |
| 太多指导等于无指导 | 当一切都"重要"，就没有什么是重要的 |
| 快速过时 | 单体手册变成过时规则的墓地 |
| 难以验证 | 单个 blob 不利于机械检查（覆盖率、新鲜度、所有权） |

### 4.3 正确做法：AGENTS.md = 目录

**新方法**：
- 不把 AGENTS.md 当百科全书，而是当作**目录**
- 知识库存在于结构化的 `docs/` 目录
- 简短的 AGENTS.md（约 100 行）注入上下文，主要作为地图，指向更深层的真实来源

### 4.4 知识库结构

**设计文档**：
- 编目和索引
- 包含验证状态
- 定义 agent-first 操作原则的核心信念集

**架构文档**：
- 提供域和包层级的顶层地图

**质量文档**：
- 评估每个产品域和架构层
- 跟踪随时间的差距

**计划作为一等公民**：
- 临时轻量计划用于小更改
- 复杂工作保存在执行计划中（包含进度和决策日志）
- 活跃计划、已完成计划、已知技术债务都版本化并共存

### 4.5 渐进式披露

使 Agent：
- 从一个小而稳定的入口点开始
- 被教导下一步去哪里
- 而不是一开始就信息过载

### 4.6 机械执行

- 专用 linter 和 CI 作业验证知识库是最新的、正确交叉链接和结构化的
- 定期的"文档园丁"Agent 扫描过时或过时的文档

## 五、架构设计

### 5.1 为 Agent 可读性优化

因为仓库完全由 Agent 生成，所以**首先为 Codex 的可读性优化**。

就像团队旨在为新工程师提高代码的可导航性一样，人类工程师的目标是让 Agent 能够直接从仓库本身推理整个业务域。

### 5.2 知识必须入库

从 Agent 的角度看，**它运行时间内无法访问的内容就等于不存在**。

- Google Docs 中的知识 → 不可访问
- 聊天线程中的知识 → 不可访问
- 人们脑海中的知识 → 不可访问
- **仓库本地的版本化产物** → 唯一能看到的

### 5.3 推入更多上下文

**关键洞察**：需要随着时间将越来越多上下文推入仓库。

- Slack 上对齐架构模式的讨论？
- 如果 Agent 发现不了，对 Agent 来说就像三个月后入职的新人不知道一样。

**正确方式**：
- 组织和暴露正确信息
- 让 Agent 可以推理
- 就像 onboarding 新团队成员一样：产品原则、工程规范、团队文化（包括 emoji 偏好）

### 5.4 技术选型偏好

倾向于：
- 可以完全内化和推理的依赖和抽象
- 被称为"无聊"的技术（由于可组合性、API 稳定性、训练集中的表现）

**案例**：
- 没有使用通用的 p-limit 包，而是自己实现了 map-with-concurrency helper
- 紧密集成 OpenTelemetry instrumentation
- 100% 测试覆盖率
- 行为完全符合运行时预期

### 5.5 强制边界，不 micromanage 实现

**核心原则**：
- 在边界强制不变量
- 不 micromanage 实现
- 让 Agent 快速发布而不破坏基础

**示例**：
- 要求 Codex 在边界解析数据形状
- 但不规定如何发生（模型似乎喜欢 Zod，但没有指定这个特定库）

### 5.6 严格架构模型

**关键洞察**：Agent 在具有严格边界和可预测结构的环境中**最有效**。

**架构规则**：
- 每个业务域分为固定的一组层
- 严格验证依赖方向
- 有限的可允许边

**层顺序**：
```
Types → Config → Repo → Service → Runtime → UI
```

**跨切面关注点**（auth, connectors, telemetry, feature flags）：
- 通过单一明确接口进入：Providers

**其他一切都被禁止，机械执行。**

### 5.7 强制执行方式

- 自定义 linter（Codex 生成！）
- 结构化测试
- 一组"品味不变量"

**品味不变量示例**：
- 静态强制结构化日志
- Schema 和类型的命名约定
- 文件大小限制
- 平台特定可靠性要求

**技巧**：因为 linter 是自定义的，可以在错误消息中注入修复指令到 Agent 上下文。

### 5.8 约束 vs 自由

- 在边界关心：正确性、可重现性
- 在边界内：允许显著自由

这类似于大型工程平台组织：**集中强制边界，本地允许自主**。

### 5.9 代码风格

**结果**：代码并不总是匹配人类风格偏好，但**没关系**。

只要输出：
- 正确
- 可维护
- 对未来 Agent 运行可读

就达到标准。

**人类品味持续反馈**：
- Review 评论
- 重构 PR
- 用户-facing bug
- 捕获为文档更新或编码到工具中

## 六、Agent 生成一切

### 6.1 Agent 生产的内容

| 类型 | 说明 |
|------|------|
| 产品代码和测试 | 应用逻辑 |
| CI 配置和发布工具 | 自动化部署 |
| 内部开发者工具 | 内部使用工具 |
| 文档和设计历史 | 设计决策记录 |
| 评估工具 | 测试和评估 |
| Review 评论和回复 | PR 审查 |
| 管理仓库本身的脚本 | 基础设施代码 |
| 生产仪表盘定义文件 | 运维配置 |

### 6.2 人类的角色

**人类始终在循环中**，但工作在不同抽象层：
- 优先级工作
- 将用户反馈转化为验收标准
- 验证结果

**当 Agent 挣扎时**：
- 视为信号：识别缺少什么（工具、护栏、文档）
- 反馈到仓库
- 始终让 Codex 自己写修复

## 七、全流程自动化

### 7.1 Agent 端到端能力

给定一个 Prompt，Agent 现在可以：

1. ✅ 验证代码库当前状态
2. ✅ 重现报告的 bug
3. ✅ 录制展示失败的视频
4. ✅ 实现修复
5. ✅ 通过驱动应用验证修复
6. ✅ 录制展示解决方案的第二个视频
7. ✅ 打开 PR
8. ✅ 响应 Agent 和人类反馈
9. ✅ 检测并修复构建失败
10. ✅ 只在需要判断时升级给人类
11. ✅ 合并更改

### 7.2 门槛

**注意**：这个行为高度依赖于这个仓库的特定结构和工具，不应该假设会普遍化——至少现在还不行。

## 八、Golden Principles（黄金原则）

### 8.1 问题：AI Slop

完全 Agent 自治也引入了新问题：
- Codex 复制仓库中已存在的模式
- 甚至是不均匀或次优的模式
- 随着时间不可避免导致漂移

### 8.2 早期方案：手动清理

- 团队过去每周五（20% 的时间）清理 "AI slop"
- **不出意料，无法扩展**

### 8.3 解决方案：黄金原则

编码"黄金原则"到仓库：

1. **优先共享工具包** vs 手写辅助函数
   - 保持不变量集中

2. **不做 YOLO-style 数据探测**
   - 验证边界或使用类型化 SDK
   - 这样 Agent 不会意外建立在猜测的形状上

### 8.4 定期清理任务

定期运行后台 Codex 任务：
- 扫描偏差
- 更新质量评级
- 打开针对性重构 PR
- 大多数可以在不到一分钟内审查并自动合并

### 8.5 像垃圾回收

- 技术债务像高息贷款
- 几乎总是持续小额偿还比让它复合然后痛苦爆发更好
- 人类品味被捕获一次，然后在每一行代码上持续执行
- 也让我们能够每天发现和解决坏模式，而不是让它们在代码库中传播数周

## 九、Conventional Norms 的改变

### 9.1 传统工程规范变得适得其反

随着 Codex 吞吐量增加，许多常规工程规范变得**适得其反**。

### 9.2 具体改变

| 传统规范 | 新做法 |
|----------|--------|
| 阻塞性合并门控 | 最小化阻塞合并门控 |
| 长寿 PR | PR 短寿 |
| 测试 flake 阻塞 | 通过后续运行解决，而不是无限阻塞 |
| 等待昂贵 | 修正便宜 |

> "在低吞吐量环境中这会是不负责任的。在这里，通常是正确的权衡。"

## 十一、总结

### 11.1 核心洞见

> "构建软件仍然需要纪律，但纪律更多地体现在**脚手架**上，而不是代码中。工具、抽象和保持代码库一致的反馈循环越来越重要。"

### 11.2 未来方向

当 Agent 如 Codex 承担更多软件生命周期工作时，问题变成：
- 如何设计环境？
- 如何设计反馈循环？
- 如何设计控制系统？

帮助 Agent 实现目标：**大规模构建和维护复杂、可靠的软件**。

## 十三、社区实践（2026年3月）

> 补充来源：字节跳动内部会议纪要

### 13.1 背景：AI Coding 的三次范式迁移

1. **第一阶段**：Cursor 等代码自动补全工具
2. **第二阶段**：Cloud Code 等 Agent Coding（agency）
3. **第三阶段**：让 Agent 完成更长时间的任务闭环

### 13.2 社区解决方案

| 方案 | 核心做法 |
|------|----------|
| **Manus** | 让模型聚焦"当前状态 + 下一步行为"，平衡记忆压缩和 prompt cache 成本 |
| **MCP (Model Context Protocol)** | 为模型提供现有内置工具无法完成任务时所需的扩展工具 |
| **Skill** | 将上下文里不具备的东西压缩成文件，配合脚本解决问题 |
| **Subagent**（Claude Code、Cursor） | 分割上下文交给子任务，解决上下文窗口有限和昂贵的问题 |
| **LSP 工具** | Agent 编码的质量底线，能给模型好的反馈，形成闭环 |
| **Midscene** | 对操作进行降维，降低上下文 TOKEN，稳定性收敛 |

### 13.3 核心方法论

**上下文管理**：
- 保持前缀稳定，让 KV Cache 高效
- 上下文膨胀时多种方式压缩
- 充分利用文件系统作为外部记忆

**工具设计原则**：
- 面向 AI：结构化 API、用自然语言取代图形
- 贴近物理上限：响应快、结构化输出
- 形成闭环：能自验证结果

### 13.4 研发者新角色

```
编码者 → 构建者

职责：
1. 设计对 AI 友好的工具
2. 管理上下文（让模型专注）
3. 构建反馈循环（让 AI 自验证）
4. 释放人类注意力
```

---

## 十四、总结：Harness Engineering 要点

1. **核心问题**：AI 写代码越多，开发者越累（测试、Review 工作指数上升）
2. **解决思路**：让 AI 完成更长时间的任务闭环
3. **关键做法**：
   - 为 Agent 构建专属工具链
   - 管理上下文，避免模型"失忆"
   - 工具要形成闭环，能自验证
4. **研发者角色**：从"编码者"转变为"构建者"

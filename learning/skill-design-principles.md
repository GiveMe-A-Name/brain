# Skill 设计原则：宪法式设计

> 来源：Anthropic "Building Effective Agents" + Block Engineering 设计原则 + Anthropic Skill 最佳实践

## 核心思想

用「宪法」替代「 SOP 」

- 不要告诉 Agent「怎么做」（步骤）
- 而是告诉 Agent「什么不能做」和「什么必须做」（约束）
- 固定 SOP 会让 Agent 在判断错误时"一条道走到黑"
- 约束式设计划定边界的同时，保留 Agent 根据上下文灵活决策的空间

---

## 五个设计原则

### 1. 二区架构：区分「一致性」和「灵活性」

| 区域 | 特点 | 例子 |
|------|------|------|
| 一致性区域 | 用脚本/硬规则（相同输入，相同输出） | 评分逻辑、验证规则、必须遵守的约束 |
| 灵活性区域 | 交给 Agent（每个 case 都不同） | 执行顺序、问题诊断路径、内容生成 |

### 2. 约束 > 步骤

给 Agent 思考的维度，而非固定的流程。

```plaintext
❌ "先检查 A，再检查 B，最后修复"
✅ "排查维度：配置、代码、知识、验证（按需选择）"
```

### 3. 写宪法，不写建议

- 越具体越好，明确约束边界
- 灰色地带用 Agent 自己判断，但红线不可逾越

### 4. 内置自我纠错机制

在 Skill 中明确写出「预警信号」：

- 修复方案没有生效
- 知识库结果与假设矛盾
- 用户反馈「还是不行」
- 超过 N 次尝试仍未解决

触发预警时，Agent 应重新评估判断，而非重复无效操作。

### 5. 简约至上 + 渐进式披露

- SKILL.md 只放入口点和核心约束（~100 行以内）
- 详细内容通过子文件按需加载
- 只添加 Agent 真正需要的上下文

---

## Anthropic 三条核心原则

1. **保持简洁** — 最简单的能工作的方案就是最好的方案
2. **保持透明** — 让 Agent 展示推理过程
3. **精心设计接口** — Agent 与工具/系统的交互方式值得投入和 prompt 工程一样多的精力

---

## 一句话总结

> 好的 Skill = 清晰的约束（宪法）+ 灵活的空间（Agent 自主决策）+ 自我纠错机制（预警信号）

# 实践验证：核心学习浓缩版（2026-03-20）

> 来源：与用户讨论 Skill 评估与优化的总结 —— 通过 Eval 验证得出的核心洞察

## 1. 隐性规则显式化

**核心洞察**：模型能从示例中学到"怎么做"（API 用法），但很难学到"什么时候该验证"以及"验证失败意味着什么"。

**举例**：
- ❌ 写"调用 getPort 返回端口号" → 低价值，看了工具定义就知道
- ✅ 写"当 buildRes.code !== 0 时，即使没抛异常也要视为失败" → 高价值，这是经验教训

**启发**：Skill 写"坑在哪里"而不是"路怎么走"。

### 与"宪法式设计"的关联

- 隐性规则就是那些"不报错但保护力弱"的约束
- 这些规则很难从示例中学习，必须显式写出
- Eval 1 的区分度正是来自这条规则：有无 `code === 0` 验证

---

## 2. 决策框架比 SOP 更有价值

**SOP 的问题**：列步骤 → 机械执行 → 缺乏判断力

**框架的优势**：给判断力
- "先判断任务类型：信息提取 → 直接读；理解意图 → 先总结；执行操作 → 先确认"
- 同样的步骤，但加上了"何时走哪条分支"的决策点

**效果**：模型少用 tokens，推理更直接。Eval 2 证明用了框架后模型少用了 10,000 tokens。

### 与"约束 > 步骤"的关联

- 决策框架 = 约束的维度
- 告诉 Agent "排查哪些维度"比告诉 Agent "先查 A 再查 B"更有价值

---

## 3. Eval 设计决定了你能看到什么

**常见误区**：设计"有 skill 就能答对"的 eval —— 这本身说明不了什么。

**好的 eval 应该**：
- 选"没有引导时模型容易滑向错误答案"的场景
- 而非"有没有 skill 都能答对"的场景

**举例**：
- ❌ 测"知不知道用 git commit" → 无区分度，所有模型都会
- ✅ 测"会不会在 commit message 里漏掉关键信息" → 才是容易犯的错

**启发**：好的测试应该淘汰人，而不是迁就人。Eval 2/3 对强模型几乎无区分度，说明 eval 本身没设计好。

---

## 4. 规划文档 ≠ Skill 草稿

**问题**：规划文档经常：
- 和其他规划文档重叠
- 假设读者已了解上下文（省略"显然"的细节）
- 写完后还要大量时间重写

**更好的方式**：
- 直接用最终格式写草稿
- 边写边问："一个完全不懂这个领域的人，看得懂吗？"
- 写完后校验："这条规则是我真正会遵守的，还是只是写出来好看的？"

---

## 核心总结

> **Skill 不是教我干活，而是帮我避坑。**
